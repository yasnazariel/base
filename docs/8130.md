# EIP-8130: Account Abstraction by Account Configuration — Implementation Plan

**Spec:** [EIP-8130](https://raw.githubusercontent.com/chunter-cb/EIPs/192db1c12780ae35bcad5d67408ea02b4adb6abb/EIPS/eip-8130.md)

EIP-8130 introduces a new EIP-2718 transaction type (`AA_TX_TYPE`) with onchain Account Configuration and external **verifier contracts** for signature verification. The protocol delegates authentication to verifiers rather than performing it natively, enabling permissionless extension of supported algorithms.

## Architecture Overview

```
┌──────────────────────────────────────────────────────────────────┐
│                  Transaction Types (base-alloy-consensus)        │
│  TxAa struct, RLP encoding, OpTxType::Aa, OpTxEnvelope::Aa     │
│  Signature payload hashes, intrinsic gas                         │
├──────────────────────────────────────────────────────────────────┤
│                  System Contracts (Solidity)                      │
│  AccountConfig.sol   @ ACCOUNT_CONFIG_ADDRESS                    │
│    - owner_config storage (verifier + scope per ownerId)         │
│    - Account creation (CREATE2)                                  │
│    - Config changes (portable + local)                           │
│    - Account lock                                                │
│    - verifySignature (ERC-1271 compatible)                       │
│  Precompiles:                                                    │
│    NonceManager      @ NONCE_MANAGER_ADDRESS (read-only)         │
│    TxContext          @ TX_CONTEXT_ADDRESS (read-only)            │
├──────────────────────────────────────────────────────────────────┤
│                  Verifier Contracts (Solidity)                    │
│  IVerifier.verify(hash, data) → ownerId                         │
│  Native: K1 (0x01), P256_RAW (0x02), P256_WEBAUTHN (0x03),     │
│          DELEGATE (0x04), Custom (0x00 + address)                │
│  Permissionless deployment, node allowlist                       │
├──────────────────────────────────────────────────────────────────┤
│                  Node Integration (Rust)                          │
│  Storage accessors (owner_config reads, nonce reads)             │
│  Validation: resolve verifier → STATICCALL → check owner_config │
│  Execution: account_changes → calls (phased atomic batching)    │
│  Auto-delegation to DEFAULT_ACCOUNT_ADDRESS for bare EOAs       │
│  Mempool: FAL invalidation, payer limits, verifier allowlist    │
└──────────────────────────────────────────────────────────────────┘
```

### Key Design Principles

1. **Verifier delegation** — The protocol does NOT verify signatures itself. It calls verifier contracts via STATICCALL. Verifiers return `ownerId`; the protocol checks `owner_config`.
2. **No public key storage** — Only `ownerId` (bytes32) + `verifier` (address) + `scope` (uint8) stored per owner. Key material provided at signing time.
3. **One slot per owner** — `owner_config` is a single 32-byte slot: `verifier (20) | scope (1) | reserved (11)`.
4. **Phased atomic calls** — `calls` field is `Call[][]` (phases of call batches). Phases commit independently; calls within a phase are atomic.
5. **Implicit EOA authorization** — Empty `owner_config` slot + `ownerId == bytes32(bytes20(account))` + K1 verifier → authorized by default.

### 1.3 — Signature Format and Payload

**EOA signature** (`from` empty): raw 65-byte ECDSA `(r || s || v)`.

**Configured owner** (`from` set): `verifier_type (1 byte) || data`
- `0x01`–`0x04`: native verifier (protocol resolves address)
- `0x00`: custom verifier (`0x00 || address (20) || data`)

| Task | Test |
|------|------|

```
deployment_code = DEPLOYMENT_HEADER(len(bytecode)) || bytecode
address = keccak256(0xff || ACCOUNT_CONFIG_ADDRESS || effective_salt || keccak256(deployment_code))[12:]
```

### 1.5 — Intrinsic Gas

```
intrinsic_gas = AA_BASE_COST + tx_payload_cost + sender_auth_cost + payer_auth_cost
              + nonce_key_cost + bytecode_cost + account_changes_cost
```

| Task | Test |
|------|------|
| `tx_payload_cost`: 16/non-zero byte + 4/zero byte over entire RLP | Standard EIP-2028 |
| `sender_auth_cost`: EOA = 6000; configured = SLOAD + verifier resolution + verifier gas | Each path |

## Milestone 4: Transaction Execution Pipeline

**Location:** `crates/alloy/eip8130-state/` + `crates/execution/evm/`

### 4.1 — Execution Order

Per the spec's [Block Execution](#block-execution):

1. If config change entries, check lock state — reject if locked
2. ETH gas deduction from payer
3. Increment nonce in Nonce Manager
4. Process `authorization_list` (EIP-7702)
5. Auto-delegate: if `code_size(from) == 0` and no create entry → set `0xef0100 || DEFAULT_ACCOUNT_ADDRESS`
6. Process `account_changes` entries:
   a. Create: register `initial_owners`, place `bytecode` at `from`
   b. Config changes: apply operations, reject if locked
7. Set transaction context on TX_CONTEXT_ADDRESS precompile
8. Execute `calls` per phase semantics (atomic within phase, phases commit independently)

| Task | Test |
|------|------|
| Gas deduction from payer (self-pay + sponsored) | Balance checks |
| Nonce increment | Storage mutation |
| Auto-delegation for bare EOAs | Code injection, skip if has code |
| Create entry: register owners + place bytecode | Owners in storage, code at address |
| Config change entries: apply ops, increment sequence | Authorize + revoke, sequence check |
| Lock check before config changes | Locked → reject |
| TX context population (sender, payer, ownerId, calls) | Precompile reads correct values |
| Call execution: single phase, multi-phase, atomic revert, phase independence | Each pattern |
| Unused gas refund to payer | Refund calculation |

## Migration from v1 Implementation

The previous implementation (branch `eip-8130-v1-backup`) was built against an earlier spec with fundamentally different abstractions. Key differences:

| Concept | v1 (old spec) | v2 (new spec) |
|---------|---------------|---------------|
| Auth model | Protocol verifies signatures natively (K1, P256, BLS) | Protocol delegates to verifier contracts |
| Key identity | `keyId` (Address, bytes20) | `ownerId` (bytes32) |
| Key storage | Multi-slot: `auth_type` + `public_key[0:32]` + `public_key[32:]` | Single slot: `verifier (20) \| scope (1) \| reserved (11)` |
| Public keys | Stored onchain | Provided at signing time |
| Signature format | `auth_type_byte \|\| ...` | `verifier_type_byte \|\| data` (verifier resolves) |
| Tx calls | `calldata` (single blob to `from`) | `calls: Call[][]` (phased atomic batching) |
| Account init | `account_initialization` field | `account_changes` typed entries |
| Key changes | `key_changes` with RLP-based format | Config change entries with ABI-encoded type hashing |
| Tx context | TSTORE on AccountConfig | TX_CONTEXT_ADDRESS precompile |
| Nonce Manager | System contract (Solidity) | Precompile (no EVM writes) |
| Fee token | `fee_token` field (reserved) | Removed |
| Payer identity | Derived from `payer_auth` ecrecover | Explicit `payer` field in tx |
| Owner scope | Not present | Bitmask: SIGNATURE, SENDER, PAYER, CONFIG |
| Account lock | Not present | Lock + timelock unlock |
| BLS | Native auth type (0x03) | Custom verifier (or future native) |
| DELEGATE ID | 0xFF | 0x04 (native verifier) |

**Reusable code from v1:**
