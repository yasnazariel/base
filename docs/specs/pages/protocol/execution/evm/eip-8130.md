# EIP-8130 Account Abstraction

## Overview

Base supports Account Abstraction (AA) transactions as typed EIP-2718 envelopes with transaction type `0x7b`.
AA transactions add:

- 2D nonce channels (`nonceKey`, `nonceSequence`)
- phased execution (`calls: Vec<Vec<Call>>`)
- optional sponsored gas payment via `payer`
- embedded sender and payer authorization payloads

## Activation

EIP-8130 behavior is activated at the `BASE_V1` hardfork.

Before `BASE_V1`:

- derivation validation rejects batches containing `0x7b` transactions
- block validation rejects blocks containing `0x7b` transactions
- txpool validation rejects incoming `0x7b` transactions

After `BASE_V1`, `0x7b` transactions are accepted end-to-end.

## RPC Surface

EIP-8130 extends existing Ethereum RPC methods:

- `eth_getTransactionCount(address, block, nonceKey?)`
  - when `nonceKey` is provided, returns the 2D nonce for `(address, nonceKey)`
  - when omitted, returns the standard account nonce
- `eth_getTransactionReceipt(hash)`
  - for `0x7b` receipts, includes `payer` and optional `phaseStatuses`

`phaseStatuses` is extracted from system logs when available and otherwise inferred from transaction outcome where possible.

## Receipt Semantics

- `payer` is the effective gas payer for the transaction.
- For self-pay EOA-mode AA transactions, `payer` is the recovered sender address (not the zero-address sentinel).
- `phaseStatuses` encodes per-phase success (`true`) or revert (`false`).
